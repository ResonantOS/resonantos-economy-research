# Tower Plan

Status: active
Promotion scope: local-research-only

## Objective

Create a governed research tower for the ResonantOS economy problem: contribution rewards, reputation, governance influence, financial-token readiness, bounty-like incentives, community growth, and exploit resistance.

## Source-First Spine

| Layer | Source basis | Research role |
| --- | --- | --- |
| L0 corpus | Official whitepaper plus CyberAlchemy v2 method evidence and DomainSpec formalization discipline. | Define what is actually being researched. |
| L1 residue map | Failure modes, incentive traps, authority risks, and missing witnesses. | Keep uncertainty visible. |
| L2 closure plan | Subtowers, dispatch proposal, gates, and next route. | Decide how to attack without over-promoting. |

## Subtower Contract

Each subtower is local research only. It may:

- collect source candidates;
- separate source claims from local readings;
- propose invariants, workflows, and risk gates;
- name bridge decisions as `borrow-carefully`, `analogy-only`, `block`, `promotion-candidate`, or `future-work`.

It may not:

- define ResonantDAO policy;
- finalize a token design;
- decide governance weights;
- create authority kinds;
- move claims into specs or implementation.
- turn CyberAlchemy v2 method authority into economy proof.

## Borrow / Block Posture

| Topic | Decision | Rationale |
| --- | --- | --- |
| Credit before money | borrow-carefully | Already aligned with the Evidence Exchange proposal and the whitepaper's Phase 1 posture. Needs anti-gaming witness before reward mechanics harden. |
| Bounty system | block-as-default | Bounties train bounty retrieval unless constrained by review, maintenance, delayed credit, and authority separation. |
| Random reward selection | analogy-only until modeled | Can deter gaming but can also feel arbitrary or unjust. Needs a formal and behavioral witness. |
| Reputation token | borrow-carefully | The whitepaper's non-transferable `$RCT` posture is useful, but scoring and governance influence need capture analysis. |
| Phase 2 algorithmic tokens | future-work | Whitepaper itself gates these on community, liquidity, treasury, products, contracts, governance, and risk controls. |

## Closure Standard

This tower is not closed when it has many papers. It is closed for the first pass when it can answer:

1. Which mechanisms are forbidden for MVP because they optimize actors toward extraction?
2. Which mechanisms are allowed only as reviewed credit signals?
3. Which invariants must hold before any Phase 2 or payout-like design is explored?
4. Which workflow witness should be tested first?
5. Which external source towers remain open and why?
